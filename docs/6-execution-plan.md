## 1. 실행계획 개요 및 원칙

- **전제**: 1인 개발, 5일 이내 MVP 완성(PRD §2.1, §8), 예산 최소화. 모든 Task 분해는 이 제약 안에서 "관리 가능한 크기"로 이뤄졌다(레이어당 5~12개, 총 32개 Task).
- **레이어 구성**: 데이터베이스(DB-xx, 8개) · 백엔드(BE-xx, 12개) · 프론트엔드(FE-xx, 12개). 각 레이어는 `4-project-structure-principles.md`에서 이미 확정된 디렉토리/파일명을 그대로 구현하는 관점에서 Task를 나눴으며, 새로운 아키텍처를 설계하지 않았다.
- **완료 조건**은 "잘 동작한다" 같은 모호한 문장이 아니라 **검증 가능한 구체적 기준**(API 응답 코드, 화면 렌더링 여부, DB 제약조건 확인 등)으로 작성했다.
- **의존성**은 같은 레이어 내 Task ID로, 다른 레이어에 대한 의존은 §7 크로스 레이어 의존성 매트릭스에서 정확한 Task ID로 정리했다.
- **범위**: 도메인 정의서 §10.1·PRD §3.1의 In-Scope를 넘어서는 Task(외부 연동, 상태 워크플로, 접근성, 낙관적 잠금 등)는 포함하지 않았다. 각 레이어 Task 목록 말미의 정합성 체크리스트(§8)로 재확인한다.

## 2. 병렬 진행 전략 — Mock API 활용

1인 개발이라도 세 레이어를 **완전히 순차적으로**(DB 끝→BE 끝→FE 끝) 진행할 필요는 없다. 프론트엔드는 `4-project-structure-principles.md` §3.3에서 이미 확정된 REST 엔드포인트 계약(URL, 메서드, 요청/응답 필드)을 기준으로 **Mock API 응답을 먼저 구현**해 백엔드 완성을 기다리지 않고 화면을 만들 수 있다. 이는 FE-02~FE-11 각 Task의 "의존성"에 "Mock API로 우선 개발 가능"으로 명시되어 있다. 백엔드가 해당 API를 완성하면 프론트엔드는 Mock을 실제 `api client` 호출로 교체만 하면 된다(구조원칙 §2.3 — api client는 hooks만 호출하는 얇은 계층이므로 교체 비용이 낮음).

DB 레이어는 API 계약과 무관하게 스키마가 먼저 확정되어야 하므로(§4 project-structure-principles.md §6.3에 테이블/컬럼이 이미 정의되어 있음) Day 1에 우선 진행하는 것을 권장한다.

- **완료 조건**:

### DB-02. 사용자/팀/팀원 마이그레이션 (users, teams, team_members)
- **설명**: `001_create_users.sql`, `002_create_teams.sql`, `003_create_team_members.sql` 세 파일을 작성해 TERM-01(User)·TERM-02(Team)·TERM-03/04(팀장/팀원 역할)를 구현한다. `team_members`는 `user_id`+`team_id` 조합과 `role`(`'leader' | 'member'`) 컬럼으로 팀 소속·역할을 표현하며, 팀장 유무는 별도 컬럼이 아니라 이 테이블의 `role` 값으로만 판단한다.
- **완료 조건**:
  - [x] `users` 테이블에 `id`, `email`(UNIQUE), `password_hash`, `created_at` 등 인증에 필요한 최소 컬럼이 존재함
  - [x] `teams` 테이블에 `id`, `name`, `created_at`이 존재하며, 팀장 여부를 나타내는 별도 컬럼(`leader_id` 등)을 두지 않음(팀장 판별은 `team_members.role`로만 수행). 초대코드 가입(FR-02)에 필요한 `invite_code`(UNIQUE) 컬럼을 추가함(swagger `Team` 스키마와 일치)
  - [x] `team_members` 테이블에 `team_id`(FK→teams), `user_id`(FK→users), `role`(`'leader'|'member'` 제약), `(team_id, user_id)` UNIQUE 제약이 존재함
  - [x] `team_members.team_id`에 인덱스가 생성됨
  - [x] 세 마이그레이션을 순서대로 실행 후 `information_schema.tables`로 `users`, `teams`, `team_members` 3개 테이블 생성이 확인됨(Windows 환경으로 `psql \dt` 대신 SQL 조회로 대체 검증)
- **의존성**: DB-01
- **관련 근거**: TERM-01/02/03/04, BR-01, BR-02, 구조원칙 §3.1 매핑표, §5.3(team_id 인덱스)
- **예상 일정**: Day 1
