## 1. 개요 및 배경

### 1.1 문제 정의 (도메인 정의서 §1 요약)

- 팀원 각자의 일정이 개인 캘린더/메신저/엑셀 등에 흩어져 있어 팀 전체 일정 파악과 충돌 방지가 어렵다.
- 일정과 그 일정에 대한 논의(대화)가 서로 다른 도구에 분리되어 있어, 일정이 왜 그렇게 정해졌는지 맥락(근거)을 되짚기 어렵다.
- 팀장이 팀 일정을 일관되게 통제하기 어려워 조율 비용이 크다.

### 1.2 목표 사용자

- **규모**: 팀당 4~10명 수준의 소규모 팀.
- **업종**: IT 개발팀, 스타트업, 소프트웨어 하우스 등 — 일정 변경이 잦고 그 변경 사유를 팀 내에서 빠르게 공유·추적해야 하는 조직.
- 상세 페르소나 카드는 본 초안 범위에서 작성하지 않으며, 위 수준의 목표 사용자 설명으로 대신한다.

### 1.3 경쟁 제품 대비 차별점

경쟁 캘린더/협업 툴 다수는 "캘린더"와 "채팅/논의"가 별도 도구이거나, 같은 제품 내에서도 서로 약하게 연결되어 있어 "이 일정이 왜 이렇게 바뀌었는가"에 대한 근거가 남지 않는다.

Team CalTalk의 핵심 차별점은 **"일정을 변경할 때 채팅 이력으로 근거를 남긴다"**는 것이다. 이는 추상적 선언이 아니라 도메인 모델 수준에서 구현된다:

- 팀원이 일정 변경이 필요할 때는 반드시 해당 일자 채팅방에서 메시지를 작성하며 **"변경 요청" 액션을 명시적으로 선택**해야 한다(TERM-08, BR-05).
- 이렇게 생성된 **일정 변경 요청(ScheduleChangeRequest)은 원본이 된 채팅 메시지와 항상 1:1로 연결**된다(도메인 정의서 §8 "일정 변경 요청과 원본 메시지의 관계").
- 팀장은 캘린더에서 특정 날짜를 선택하면 해당 일자의 채팅 이력(변경 요청 포함)을 함께 확인할 수 있고(BR-07, UC-07), 그 맥락을 바탕으로 일정을 직접 수정한다(UC-06).

즉, "일정 변경의 근거가 채팅 이력에 항상 남는다"는 차별점은 자연어 분석 등 부가 기능이 아니라, **ScheduleChangeRequest ↔ ChatMessage의 1:1 연결이라는 단순하고 결정론적인 데이터 모델**로 구현된다. 이 단순함이 1인 개발·5일 MVP라는 제약 안에서도 핵심 차별점을 구현 가능하게 만드는 요인이다.

## 2. 목표 및 성공 지표

### 2.1 프로젝트 목표

| 구분 | 목표 |
|---|---|
| 제품 목표 | 팀 캘린더와 팀 채팅을 한 화면에서 제공하고, 일정 변경에는 채팅 근거가 남도록 한다. |
| 비즈니스 목표 | **3,000개 이상의 팀이 동시에 프로젝트를 진행**할 수 있는 서비스로 성장 가능한 기반을 만든다 (팀당 4~10명 기준, 이론상 최대 약 3만 명 규모의 동시 사용자를 감안한 확장 목표). |
| 프로젝트 관리 목표 | **1인 개발, 5일 이내에 핵심 기능(MVP) 완성**. |

> **중요한 전제**: "3,000+ 팀 동시 지원"은 서비스가 목표로 하는 **성장 단계의 목표치**이며, 5일 MVP는 이 목표를 위한 **핵심 기능의 완성**을 의미한다. 3,000+ 팀 규모의 실제 부하를 5일 안에 인프라적으로 검증·확보한다는 뜻이 아니다. 이 둘 사이의 관계와 리스크는 §9(리스크 및 향후 로드맵)에서 명시적으로 다룬다.

### 2.2 성공 지표 (MVP 기준)

| 지표 | 목표 | 비고 |
|---|---|---|
| MVP 기능 완성 | 5일 내 §5 "Must" 요구사항 전체 동작 | 1인 개발 기준 |
| 핵심 유스케이스 커버리지 | 도메인 정의서 UC-01~UC-07 전부 시연 가능 | 회귀 시나리오로 활용 |
| 일정-채팅 연동 사용성 | 캘린더에서 날짜 선택 → 해당 일자 채팅 이력 조회까지 3클릭 이내 | BR-07 구현 확인 |
| 변경요청-반영 흐름 동작률 | "변경 요청" 액션으로 생성된 ScheduleChangeRequest가 팀장 화면에 100% 노출 | BR-05, UC-06 |
| 응답 속도(단일 인스턴스 기준) | 주요 API(캘린더 조회/일정 CRUD/메시지 전송) 평균 응답 500ms 이내 | 소규모 팀(4~10인) 동시 사용 기준, 부하테스트 아님 |
| 확장성 설계 목표(향후) | 팀 수 3,000+, 동시 사용자 최대 약 3만 명 규모를 상정한 아키텍처 확장 여지 확보 | MVP 단계는 "막지 않는 설계"까지만, 실측 검증은 로드맵(§9) |

## 3. 범위 (Scope)

도메인 정의서 §10과 정합성을 유지하되, PRD 관점에서 "기능" 단위로 재정리한다.

### 3.1 In-Scope (MVP 범위)

- 사용자 인증(로그인/회원가입) 및 인증 기반 접근 제어 (BR-01)
- 팀 생성 시 생성자의 팀장 자동 임명 등 최소 팀 라이프사이클 (도메인 정의서 §3.3)
- 팀 단위 캘린더의 월/주/일 조회 (BR-08, UC-01)
- 팀장의 일정 생성/수정/삭제 (BR-03, UC-02, UC-03)
- 팀원의 일정 조회 전용 접근 (BR-04)
- 일자별 채팅방 및 채팅 이력 (BR-06, UC-04)
- 채팅 메시지 작성 시 "일반 메시지" / "변경 요청" 명시적 구분, 이에 따른 ScheduleChangeRequest 생성 (BR-05, TERM-08, UC-05)
- 팀장의 변경 요청 검토 및 수동 일정 반영 (UC-06, 상태 머신 없음)
- 캘린더-채팅 연동 조회(날짜 선택 시 해당 일자 채팅 이력 표시) (BR-07, UC-07)
- 다중 팀 소속 시 팀 전환 방식의 캘린더 조회 (BR-11)
- 동시 수정 시 Last-Write-Wins 처리 (BR-10, 별도 락 없음)
- 반응형 웹 UI (데스크톱/모바일 브라우저 대응, 네이티브 앱 아님)

### 3.2 Out-of-Scope (도메인 정의서 §10.2와 동일하게 유지)

- 외부 캘린더/메신저 연동 (Google Calendar, Slack, Outlook 등) — 로드맵으로 이관 (§9)
- 결제/구독 등 과금 모델
- 다중 팀장 체계, 부팀장 등 세부 권한 커스터마이징
- ScheduleChangeRequest의 승인/반려/보류 등 정식 상태 전이 워크플로 및 처리 결과 알림
- 팀 초대·가입·팀장 이양의 상세 UX/절차
- 낙관적 잠금 등 정교한 동시성 제어
- 사용자 로컬 타임존 기준 채팅방 일자 구분 (서버 UTC 자정 기준 고정)
- 푸시/이메일 등 외부 알림 채널
- 모바일 네이티브 앱, 오프라인 동기화
- 반복 일정(recurring schedule)
- 접근성(A11y) 대응
- 다국어/국제화

## 4. 사용자 및 역할

상세 페르소나는 작성하지 않으며, 도메인 정의서의 액터 정의를 그대로 인용한다.

### 4.1 팀장 (Team Leader, TERM-03)
- 팀 일정의 생성·수정·삭제 권한 보유 (BR-03).
- 팀원이 남긴 일정 변경 요청을 검토하고, 필요 시 일정을 직접 수정하여 반영 (UC-06).
- 팀 캘린더 및 채팅 이력 조회 가능.
- 한 팀에 팀장은 1명 (다중 팀장/부팀장은 범위 외).

### 4.2 팀원 (Team Member, TERM-04)
- 팀 일정 조회만 가능, 직접 CUD 불가 (BR-04).
- 일정 변경이 필요하면 해당 일자 채팅방에서 "변경 요청" 액션을 명시적으로 선택하여 요청 (BR-05, UC-05).
- 채팅에 참여하여 메시지 송수신 가능.

### 4.3 목표 사용자 규모/맥락

4~10인 규모의 팀(IT 개발팀, 스타트업, 소프트웨어 하우스)이 각자의 팀 공간 안에서 팀장/팀원 역할로 위 기능을 사용한다고 가정한다. 한 사용자가 팀 A에서는 팀장, 팀 B에서는 팀원일 수 있다(역할은 팀 소속 관계에 종속, 도메인 정의서 §3.2).

## 5. 핵심 기능 요구사항 (Functional Requirements)

도메인 정의서의 UC-xx/BR-xx를 기능 단위로 그룹핑하고 MoSCoW 우선순위를 부여한다. **5일 MVP는 Must 항목 전체 완성을 목표로 하며, Should/Could는 여유가 있을 때만 시도한다.**

| FR ID | 기능 요구사항 | 우선순위 | 근거(도메인 정의서) |
|---|---|---|---|
| FR-01 | 이메일 기반 회원가입/로그인, 인증 기반 접근 제어 | **Must** | BR-01, TERM-01 |
| FR-02 | 팀 생성(생성자=팀장 자동 임명), 초대코드/링크를 통한 팀 가입 | **Must** | §3.3, BR-02 |
| FR-03 | 팀 캘린더 월/주/일 뷰 조회 (소속 팀 한정) | **Must** | BR-08, UC-01 |
| FR-04 | 팀장의 일정 생성 (제목 1~100자, 시작<종료, 대상=팀전체/특정팀원 1인 단일선택) | **Must** | BR-03, UC-02, §8 |
| FR-05 | 팀장의 일정 수정/삭제 | **Must** | BR-03, UC-03 |
| FR-06 | 팀원의 일정 조회 전용 UI (CUD 버튼 비노출) | **Must** | BR-04 |
| FR-07 | 일자별 채팅방 자동 매핑 및 채팅 이력 누적 (서버 UTC 자정 기준 일자 구분) | **Must** | BR-06, TERM-06/07/09, §8 |
| FR-08 | 채팅 메시지 작성 시 "일반 메시지" / "변경 요청" 명시적 선택 UI, 1~500자 검증 | **Must** | BR-05, TERM-07/08, §8 |

| FR-12 | 캘린더에서 날짜 선택 시 해당 일자 채팅 이력을 같은 화면(또는 연동 패널)에서 조회 | **Must** | BR-07, UC-04, UC-07 |
| FR-13 | 다중 팀 소속 시 팀 선택(전환) UI로 캘린더 조회 | **Should** | BR-11 |

| FR-15 | 팀장 탈퇴 시 "다른 팀장 지정 전에는 탈퇴 불가" 최소 검증(차단 메시지 수준) | **Should** | BR-09, §6.1 |

| FR-18 | 팀 초대 UX 고도화(승인 절차, 이메일 인증 등) | **Could / 로드맵** | §10.2, 범위 외 |
| FR-19 | 동시수정 충돌 시 경고/병합 UI, 낙관적 잠금 | **Could / 로드맵** | BR-10, §10.2 |

> FR-18, FR-19는 도메인 정의서 §10.2에서 이미 범위 외로 명시된 항목이며, 본 PRD에서도 5일 MVP 범위에 포함하지 않는다(§9 로드맵 참고).

## 6. 비기능 요구사항 (Non-Functional Requirements)

| 구분 | 요구사항 |
|---|---|
| **성능(MVP 기준)** | 4~10인 팀 규모의 동시 사용 환경에서 주요 API(캘린더 조회, 일정 CRUD, 메시지 전송) 평균 응답 500ms 이내를 목표로 한다. 이는 부하 테스트로 검증된 SLA가 아니라 개발 중 체감 목표치다. |
| **확장성(설계 목표, 검증은 로드맵)** | 비즈니스 목표인 "3,000개 이상 팀 동시 지원"을 염두에 두고, 팀/일정/메시지 데이터를 팀 ID 기준으로 분리 조회 가능하도록 스키마를 설계한다(예: 모든 주요 테이블에 team_id 인덱스). 다만 MVP 단계에서 실제 3,000+ 팀 규모의 부하 테스트나 인프라 스케일링(캐싱, 커넥션 풀 튜닝, 수평 확장 등)은 수행하지 않으며, §9에서 향후 과제로 명시한다. |
| **가용성** | 별도 SLA를 정의하지 않는다(1인 개발, 예산 최소화 전제). 단일 인스턴스 운영을 기본 가정으로 한다. |
| **보안** | 비밀번호 해시 저장, HTTPS 사용, 인증 토큰(JWT 또는 세션 쿠키) 기반 접근 제어, 팀 데이터 접근 시 소속 팀 검증(BR-02). 정교한 보안 감사/침투테스트는 범위 외. |
| **플랫폼/UI** | 웹 우선(Web-first). 데스크톱 브라우저를 기본으로 하고, 모바일 브라우저에서도 반응형 레이아웃으로 캘린더/채팅을 사용할 수 있도록 한다. 네이티브 모바일 앱은 범위 외. |
| **접근성(A11y)** | 이번 프로젝트에서 고려하지 않는다(명시적 범위 외). |
| **국제화** | 한국어 단일 언어로 제공. 다국어는 범위 외. |
| **데이터 정합성** | 도메인 정의서 §8의 경계 조건(제목 길이, 시작<종료, 메시지 길이, 일자 구분 기준 등)을 서버 측 검증 규칙으로 구현한다. |
| **동시성** | BR-10에 따라 Last-Write-Wins를 기본 정책으로 하며, 낙관적 잠금 등은 도입하지 않는다. |

## 7. 기술 아키텍처 개요

과설계를 피하고, 주어진 스택(React + Node.js/Express + PostgreSQL) 안에서 5일 내 구현 가능한 수준으로만 개요를 제시한다.

```
[React SPA (반응형 웹)]
        │  HTTPS / REST API (JSON)
        ▼
[Node.js + Express API 서버]
   - 인증(JWT 또는 세션)
   - 팀/일정/채팅/변경요청 라우트
   - 요청 시 팀 소속 검증(BR-02) 미들웨어
        │  SQL (예: node-postgres/pg 또는 경량 ORM)
        ▼
[PostgreSQL]
   - users, teams, team_members(역할 포함)
   - schedules
   - chat_rooms(team_id + date) 또는 date 컬럼으로 논리적 구분
   - chat_messages(is_change_request 플래그 등)
   - schedule_change_requests(chat_message_id 1:1 FK)
```

- **프론트엔드**: React 단일 페이지 애플리케이션. 캘린더(월/주/일 뷰)와 채팅 패널을 한 화면에 배치하고, 미디어 쿼리 기반 반응형 레이아웃 적용.
- **백엔드**: Node.js + Express 기반 REST API. 인증 미들웨어에서 BR-01(인증 필수)·BR-02(팀 소속 필수)를 공통 처리.
- **데이터베이스**: PostgreSQL 단일 인스턴스. team_id를 기준으로 한 인덱스 설계로 향후 팀 수 증가에 대한 조회 성능 확장 여지를 남긴다(실제 확장 작업은 §9 로드맵).
- **배포/인프라**: 1인 개발·최소 예산 전제에 따라 무료/저비용 티어(PaaS형 호스팅 등)를 활용하는 것을 가정하며, 별도의 이중화·오토스케일링 인프라는 MVP 단계에 포함하지 않는다.
- **외부 연동**: 없음. Google Calendar, Slack 등 외부 서비스 연동은 이번 범위에 포함하지 않는다(§10.2, §9 로드맵).

## 8. 일정 계획 (5일 MVP, 1인 개발 기준)

전제: 개발자 1인, 5일 이내 핵심 기능(Must 항목) 완성. 아래는 대략적인 Day-by-day 마일스톤이며, 실제 진행 상황에 따라 순서는 조정될 수 있다.

| Day | 마일스톤 | 포함 FR |
|---|---|---|
| Day 1 | 프로젝트 뼈대 구성, DB 스키마 설계·마이그레이션, 회원가입/로그인, 팀 생성/가입(초대코드 수준) | FR-01, FR-02 |
| Day 2 | 팀 캘린더 월/주/일 조회 UI, 팀장의 일정 생성/수정/삭제, 팀원 조회 전용 뷰 | FR-03, FR-04, FR-05, FR-06 |
| Day 3 | 일자별 채팅방/채팅 이력, 메시지 전송(일반/변경요청 구분 UI), ScheduleChangeRequest 생성 로직 | FR-07, FR-08, FR-09 |
| Day 4 | 팀장용 변경요청 확인 화면, 수동 반영 플로우, 캘린더-채팅 연동 조회, 다중 팀 전환 UI | FR-10, FR-11, FR-12, FR-13 |

- **Must 항목(§5)**: Day 1~4에 걸쳐 우선 구현하며, Day 5는 버퍼(안정화·배포)로 확보한다.
- **Should 항목(FR-13~16)**: 여유가 있는 경우 Day 4~5에 반영하고, 시간이 부족하면 다음 스프린트로 이월한다.
- **Could/로드맵 항목(FR-18~19 등)**: 5일 MVP 범위에서 명시적으로 제외한다.

## 9. 리스크 및 향후 로드맵

### 9.1 주요 리스크

| 리스크 | 내용 | 대응 방향 |
|---|---|---|
| 1인 개발 리스크 | 개발자 1인이므로 일정 지연, 리뷰 부재로 인한 품질 저하, 병목(버스 팩터=1) 위험이 있다. | 기능 범위를 도메인 정의서 §10.1 수준으로 타이트하게 유지하고, Should/Could 항목을 과감히 다음 단계로 미룬다. |
| 5일 일정 리스크 | 채팅-캘린더 연동, 변경요청 흐름 등 여러 기능이 얽혀 있어 통합 테스트 시간이 부족할 수 있다. | Day 5를 버퍼로 명시적으로 확보하고, Must 항목만으로도 UC-01~07 전체가 동작하도록 우선순위를 관리한다. |
| **"3,000+ 팀 동시 지원" 목표와 5일 MVP·1인 개발·PostgreSQL 단일 스택 사이의 긴장** | 3,000+ 팀(최대 약 3만 사용자) 규모를 실제로 지탱하려면 커넥션 풀링 튜닝, 캐싱, 읽기 복제본, 수평 확장 등의 인프라 작업이 필요하지만, 이는 5일·1인 개발로 동시에 달성하기 어렵다. | **5일 MVP는 "3,000+ 팀이 써도 기능적으로 올바르게 동작하는 로직"을 완성하는 데 집중**하고, 실제 대규모 동시 접속 처리 능력(인프라 스케일링·부하 검증)은 MVP 이후 별도 단계에서 검증한다. 스키마 설계 시 team_id 인덱싱 등 "확장을 막지 않는" 선택만 MVP 단계에 반영한다. |
| 동시성/경합 | Last-Write-Wins로 인한 의도치 않은 덮어쓰기 가능성(BR-10). | MVP에서는 허용된 리스크로 수용하고, 사용자에게는 별도 경고 없이 최신 저장이 반영됨을 전제로 한다. 낙관적 잠금은 로드맵으로 이관. |
| 예산 제약 | 유료 SaaS/인프라 사용을 최소화해야 한다. | 무료 티어/오픈소스 중심 구성으로 운영 비용을 낮추고, 유료 확장은 실제 팀 수 증가 시점에 검토한다. |

### 9.2 향후 로드맵 (MVP 이후, 도메인 정의서 §10.2 기반)

- **외부 서비스 연동**: Google Calendar, Slack 등과의 연동(가져오기/내보내기, 알림 연계).
- **정식 변경요청 워크플로**: ScheduleChangeRequest에 대한 승인/반려/보류 상태, 처리 결과 알림, 처리 이력 추적.
- **동시성 제어 고도화**: 낙관적 잠금 등 정교한 충돌 방지/병합 UX.
- **일자 구분 기준 개선**: 서버 UTC 자정 고정 방식에서 사용자 로컬 타임존 기준으로 전환 검토.
- **팀 라이프사이클 UX 고도화**: 초대 승인 절차, 팀장 이양 UX, 다중 팀장/부팀장 등 세부 권한 모델.
- **반복 일정**, **푸시/이메일 알림**, **모바일 네이티브 앱**, **접근성(A11y) 대응**, **다국어 지원**.
- **인프라 확장성 검증**: 3,000+ 팀·최대 약 3만 동시 사용자 목표에 대한 실제 부하 테스트, 캐싱 레이어, DB 커넥션 풀링/읽기 복제본, 필요 시 수평 확장 아키텍처 검토.
