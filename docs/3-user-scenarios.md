## 1. 작성 원칙 및 전제

### 1.1 상세 페르소나를 작성하지 않는 이유

PRD §4는 "상세 페르소나 카드는 본 초안 범위에서 작성하지 않으며, 위 수준의 목표 사용자 설명으로 대신한다"고 명시적으로 결정했다. 본 시나리오 문서도 이 결정을 그대로 따른다. 즉, 이름·나이·직업·취미 등 허구적 배경 서사를 가진 인물 카드는 만들지 않고, **역할(팀장/팀원) 기반의 간결한 표기**만 사용한다.

### 1.2 역할 표기 규칙

| 표기 | 의미 |
|---|---|
| 팀장 | 해당 팀의 팀장(TERM-03). 시나리오에 팀장이 1명만 등장하면 그냥 "팀장"으로 표기한다. |
| 팀원A, 팀원B | 해당 팀의 팀원(TERM-04). 한 시나리오에 여러 팀원이 필요할 때 구분을 위해 A/B를 붙인다. 특정 개인을 지칭하는 이름이 아니다. |
| 사용자 | 아직 인증 전이거나, 역할이 시나리오 맥락상 중요하지 않은 경우에 사용한다. |

한 사용자가 여러 팀에 속할 경우 팀에 따라 역할이 달라질 수 있다는 점(도메인 정의서 §3.2)을 시나리오 작성 시 전제로 둔다.

### 1.3 추적성 원칙

- 모든 핵심 시나리오는 관련 UC-ID(도메인 정의서 §7), BR-ID(도메인 정의서 §5), FR-ID(PRD §5)를 명시한다.
- 시나리오 단계는 "사용자 행동 → 시스템 반응" 쌍으로 서술하여, 이후 화면 설계 및 QA 테스트케이스 작성 시 그대로 절차 기반 테스트 스텝으로 옮길 수 있도록 한다.
- 도메인 정의서/PRD에 정의되지 않은 세부 UX(정확한 에러 문구, 버튼 위치, 색상 등)는 이 문서에서 새로 창작하지 않는다. 필요한 경우 "차단 메시지를 노출한다"와 같이 동작 수준까지만 서술하고, 문구·비주얼 디자인은 화면 설계 단계로 넘긴다.

### 1.4 시나리오 규모

5일 MVP·1인 개발이라는 제약(PRD §8, §9.1)을 고려하여, 핵심 시나리오(Happy Path)는 9개, 예외 시나리오는 4개로 한정한다. 이는 도메인 정의서 UC-01~07 전부와 BR-09~12(§6 예외 상황) 전부를 커버하는 최소 집합이다.

## 2. 핵심 시나리오 (Happy Path)

### SC-01. 로그인 후 팀 캘린더 조회

- **관련 ID**: UC-01 / BR-01, BR-02, BR-08 / FR-01, FR-03

**사전 조건**: 사용자는 이미 회원가입되어 있고, 최소 1개 팀에 소속되어 있다.

**단계별 흐름**
1. 사용자가 로그인 화면에서 이메일/비밀번호를 입력하고 "로그인" 버튼을 클릭한다. → 시스템은 인증 정보를 검증하고 세션(또는 JWT)을 발급한다.
2. 로그인 성공 시 시스템은 사용자가 소속된 팀 중 기본 팀(예: 최근 접속 팀 또는 첫 번째 팀)의 캘린더 화면으로 이동시킨다.
3. 사용자는 캘린더 상단의 뷰 전환 컨트롤에서 "월" 보기를 선택한다. → 시스템은 해당 팀의 이번 달 일정을 캘린더 그리드에 표시한다.
4. 사용자가 "주" 보기로 전환한다. → 시스템은 같은 팀의 일정을 주 단위로 재구성해 표시한다.
5. 사용자가 "일" 보기로 전환한다. → 시스템은 선택된 하루의 일정만 표시한다.

**기대 결과 / 성공 기준**: 사용자는 로그인한 팀에 한해(BR-02) 월/주/일 세 가지 뷰로 일정을 오류 없이 조회할 수 있다. 비로그인 상태이거나 소속되지 않은 팀의 일정은 조회되지 않는다(BR-01, BR-02).

---

### SC-02. 팀장의 신규 팀 일정 등록

- **관련 ID**: UC-02 / BR-03 / FR-04

**사전 조건**: 팀장이 팀 캘린더 화면에 로그인되어 있다.

**단계별 흐름**
1. 팀장이 캘린더에서 일정을 등록할 날짜(또는 "일정 추가" 버튼)를 클릭한다. → 시스템은 일정 생성 폼을 표시한다.
2. 팀장이 제목(1~100자)을 입력한다.
3. 팀장이 시작 일시와 종료 일시를 입력한다(시작 < 종료).
4. 팀장이 배정 대상으로 "팀 전체" 또는 "특정 팀원 1명" 중 하나를 선택한다(단일 선택).
5. 팀장이 "저장"을 클릭한다. → 시스템은 입력값을 검증(제목 길이, 시작<종료, 대상 단일 선택 여부)하고, 통과하면 새 일정을 생성해 캘린더에 반영한다.

**기대 결과 / 성공 기준**: 새 일정이 팀 캘린더(월/주/일 뷰 모두)에 즉시 표시되며, 해당 팀 소속 팀원들도 조회 시 동일한 일정을 볼 수 있다. 검증 실패 시(예: 시작≥종료) 시스템은 저장을 차단하고 오류를 알린다.

---

### SC-03. 팀장의 기존 일정 수정/삭제

- **관련 ID**: UC-03 / BR-03 / FR-05

**사전 조건**: 팀 캘린더에 팀장이 등록한 일정이 최소 1건 존재한다.

**단계별 흐름 (수정)**
1. 팀장이 캘린더에서 기존 일정을 클릭한다. → 시스템은 일정 상세/수정 폼을 표시한다.
2. 팀장이 제목, 시작/종료 일시, 배정 대상 중 하나 이상을 변경한다.
3. 팀장이 "저장"을 클릭한다. → 시스템은 값을 검증하고 해당 일정 레코드를 갱신한 뒤 캘린더에 즉시 반영한다.

**단계별 흐름 (삭제)**
4. 팀장이 일정 상세 화면에서 "삭제"를 클릭한다. → 시스템은 삭제 확인을 요청한다.
5. 팀장이 삭제를 확정한다. → 시스템은 해당 일정을 캘린더에서 제거한다.

**기대 결과 / 성공 기준**: 수정된 내용 또는 삭제 결과가 팀 캘린더에 즉시 반영되며, 팀원 화면에서도 동일하게 보인다. 팀원 계정으로는 수정/삭제 액션 자체가 노출되지 않는다(BR-04, FR-06).

---

### SC-04. 팀원의 조회 전용 캘린더 이용

- **관련 ID**: UC-01 / BR-04 / FR-06

**사전 조건**: 팀원A가 로그인되어 있고, 소속 팀에 팀장이 등록한 일정이 존재한다.

**단계별 흐름**
1. 팀원A가 캘린더 화면에 접속한다. → 시스템은 소속 팀의 일정을 월/주/일 뷰로 표시한다.
2. 팀원A가 일정 항목을 클릭한다. → 시스템은 일정 상세 정보(제목/일시/대상)를 읽기 전용으로 표시한다.
3. 팀원A는 화면에서 일정 생성/수정/삭제 버튼을 찾지만, 시스템은 해당 버튼을 팀원 계정에는 노출하지 않는다.

**기대 결과 / 성공 기준**: 팀원A는 팀 일정을 자유롭게 조회할 수 있으나, 어떤 경로로도 직접 CUD를 수행할 수 없다. 일정 변경이 필요하면 SC-06(변경 요청)으로 이어진다.

---

### SC-05. 팀원이 캘린더에서 날짜를 선택해 채팅 이력을 확인

- **관련 ID**: UC-04 / BR-06, BR-07 / FR-07, FR-12

**사전 조건**: 해당 날짜 채팅방에 이전에 주고받은 메시지가 존재한다.

**단계별 흐름**
1. 팀원A가 캘린더에서 특정 날짜 셀을 클릭한다. → 시스템은 해당 날짜에 대응하는 채팅방(팀ID + 날짜로 식별)을 연동 패널(또는 같은 화면)에 연다.
2. 시스템은 그 날짜에 누적된 채팅 이력을 시간순으로 표시한다.
3. 팀원A가 채팅 이력을 스크롤하며, 일반 메시지와 변경 요청 메시지가 구분되어 표시되는 것을 확인한다.

**기대 결과 / 성공 기준**: 사용자는 캘린더 날짜 선택에서 채팅 이력 확인까지 최소한의 클릭(PRD 목표: 3클릭 이내)으로 도달할 수 있고, 이력은 날짜별로 분리되어 보존된다(BR-06).

### SC-06. 팀원이 "변경 요청" 액션으로 일정 변경을 요청 (핵심 차별점)

- **관련 ID**: UC-05 / BR-05, TERM-07, TERM-08 / FR-08, FR-09

이 시나리오는 Team CalTalk의 핵심 차별점(PRD §1.3) — "일정 변경에는 채팅 이력으로 근거가 남는다" — 을 구현하는 흐름이다. 핵심은 **"일반 메시지"와 "변경 요청"이 메시지 작성 시점에 명시적으로 구분되어 전송된다는 것**과, 그렇게 생성된 ScheduleChangeRequest가 **원본 채팅 메시지와 항상 1:1로 연결된다는 것**이다.

**사전 조건**: 팀원A가 특정 날짜의 채팅방을 열람 중이며, 해당 날짜(또는 인접일)에 자신이 관련된 일정이 존재한다.

**단계별 흐름**
1. 팀원A가 채팅 입력창에 "다음 주 화요일 회의를 3시로 옮겼으면 합니다"라는 문구를 입력한다.
2. 팀원A는 메시지 전송 전 UI에서 두 가지 전송 유형 중 하나를 명시적으로 선택해야 한다: **[일반 메시지]** 또는 **[변경 요청]**(TERM-07). 팀원A는 **[변경 요청]**을 선택한다.
3. 시스템은 팀원A에게 이 메시지가 대상으로 하는 일정 관련 정보(예: 신규/수정/취소 중 요청 유형, 관련 일정)를 함께 지정하도록 요구한다(도메인 정의서 §4 관련 서술).
4. 팀원A가 "전송"을 클릭한다. → 시스템은 메시지 본문 길이(1~500자)를 검증한다.
5. 검증 통과 시, 시스템은 다음 두 레코드를 함께 생성한다:
   - 채팅 메시지(ChatMessage): 발신자=팀원A, 유형=변경 요청, 날짜=해당 채팅방 날짜.
   - 일정 변경 요청(ScheduleChangeRequest): 위 채팅 메시지와 **1:1로 연결**, 요청 의도(신규/수정/취소) 포함.
6. 시스템은 채팅 이력에 팀원A의 메시지를 "변경 요청"임을 시각적으로 구분해(예: 일반 메시지와 다른 표기) 표시한다.
7. (대비를 위한 참고 흐름) 만약 같은 화면에서 팀원B가 동일한 채팅방에 "저도 그날 참석 가능합니다"라는 메시지를 **[일반 메시지]**로 선택해 전송하면, 시스템은 이를 일반 ChatMessage로만 저장하고 ScheduleChangeRequest는 생성하지 않는다(BR-05). 메시지 본문에 "일정", "변경" 같은 단어가 포함되어도 자동으로 변경 요청으로 판별하지 않는다.

**기대 결과 / 성공 기준**: "변경 요청"으로 명시적으로 전송된 메시지에 대해서만 ScheduleChangeRequest가 100% 생성되고(PRD 성공지표), 생성된 요청은 원본 채팅 메시지와 1:1 관계를 유지한다. 일반 메시지로 전송된 내용은 아무리 일정 변경을 암시하더라도 요청으로 취급되지 않는다.

### SC-07. 팀장이 변경 요청을 검토하고 일정에 직접 반영 (핵심 차별점, SC-06과 연결)

- **관련 ID**: UC-06 / BR-05 / FR-10, FR-11

이 시나리오는 SC-06에서 생성된 ScheduleChangeRequest를 팀장이 어떻게 확인하고 처리하는지를 다룬다. **주의**: 팀장은 요청에 "승인/반려" 같은 상태를 부여하지 않는다. 검토 후 필요하다고 판단되면 해당 일정을 직접 수정하는 것으로 처리가 끝난다(도메인 정의서 §1.2, PRD §3.2에서 정식 상태 워크플로를 범위 외로 규정).

**사전 조건**: SC-06에서 팀원A가 "변경 요청"을 전송해 ScheduleChangeRequest가 생성되어 있다.

**단계별 흐름**
1. 팀장이 로그인 후 팀 화면에서 미반영 변경 요청 목록(또는 표시)을 확인한다. → 시스템은 아직 반영되지 않은 ScheduleChangeRequest들을 노출한다.
2. 팀장이 목록에서 팀원A의 요청 항목을 클릭한다. → 시스템은 해당 요청이 연결된 **원본 채팅 메시지**(팀원A가 작성한 "다음 주 화요일 회의를 3시로 옮겼으면 합니다")와 요청 맥락(날짜, 요청 유형)을 함께 보여준다.
3. 팀장은 필요 시 해당 날짜 채팅방으로 이동해 앞뒤 대화 맥락(예: 팀원B의 참석 가능 메시지 등)을 추가로 확인한다.
4. 팀장이 맥락을 검토한 뒤, 반영이 타당하다고 판단하여 캘린더에서 해당 일정을 클릭하고 시작 시간을 15:00으로 직접 수정한 뒤 저장한다(SC-03의 일정 수정 흐름과 동일한 경로).
5. 시스템은 일정 레코드를 갱신하고 캘린더에 즉시 반영한다. ScheduleChangeRequest 레코드 자체에는 상태값 변경이 발생하지 않는다(도메인 정의서 §1.2).

**기대 결과 / 성공 기준**: 팀장은 채팅 메시지 원문을 근거로 삼아 일정을 직접 수정할 수 있고, 팀원A와 다른 팀원들은 이후 채팅 이력과 캘린더를 통해 "왜 이 일정이 이렇게 바뀌었는지"를 그대로 되짚을 수 있다(PRD §1.3 차별점 실현).

---

### SC-08. 팀원이 캘린더와 채팅을 한 화면에서 동시에 확인

- **관련 ID**: UC-07 / BR-07 / FR-12

**사전 조건**: 팀원A가 소속 팀의 캘린더/채팅 화면에 접속해 있다.

**단계별 흐름**
1. 팀원A가 캘린더에서 다가오는 회의 일정이 있는 날짜를 클릭한다.
2. 시스템은 같은 화면 내에서 캘린더 영역과 해당 날짜의 채팅 패널을 나란히(또는 반응형 레이아웃에서는 전환 가능한 형태로) 표시한다.
3. 팀원A는 캘린더에 표시된 일정 정보를 보면서, 동시에 채팅 패널에서 그 일정과 관련된 논의(예: SC-06/SC-07에서 오간 변경 요청과 팀장의 반영 결과)를 함께 확인한다.

**기대 결과 / 성공 기준**: 사용자는 화면 전환 없이(또는 최소한의 전환으로) 일정과 그 논의 맥락을 함께 파악할 수 있다. 데스크톱/모바일 브라우저 모두에서 반응형으로 동작한다(FR-17).

---

### SC-09. 여러 팀에 소속된 사용자의 팀 전환 조회

- **관련 ID**: UC-01 / BR-11 / FR-13

**사전 조건**: 사용자가 팀 A(팀원 역할)와 팀 B(팀장 역할)에 동시에 소속되어 있다.

**단계별 흐름**
1. 사용자가 로그인한다. → 시스템은 기본 팀(예: 마지막 접속 팀)의 캘린더를 표시한다. 가정상 팀 A가 기본으로 열린다.
2. 사용자는 팀 A 캘린더에서 팀원으로서 일정을 조회만 할 수 있음을 확인한다(SC-04와 동일한 조회 전용 화면).
3. 사용자가 화면 상단의 팀 선택 UI를 클릭하고 팀 B를 선택한다. → 시스템은 화면을 팀 B의 캘린더로 전환한다.
4. 팀 B 화면에서는 사용자가 팀장 역할이므로 일정 생성/수정/삭제 버튼이 노출된다.
5. 사용자가 다시 팀 A로 전환한다. → 시스템은 팀 A 캘린더로 되돌아가며, 두 팀의 일정이 한 화면에 병합되어 표시되지 않는다.

**기대 결과 / 성공 기준**: 사용자는 한 번에 하나의 팀을 선택해 그 팀의 캘린더만 조회하며(BR-11), 팀에 따라 자신의 역할(팀장/팀원)에 맞는 화면과 권한이 정확히 적용된다.

## 3. 예외/에러 시나리오

### EX-01. 팀장 탈퇴 시도 (팀장 부재 방지, BR-09 / §6.1)

**상황**: 팀장이 자신이 유일한 팀장인 팀에서 탈퇴를 시도한다.

**흐름**
1. 팀장이 팀 설정 화면에서 "팀 탈퇴"를 클릭한다.
2. 시스템은 해당 팀에 다른 팀장이 지정되어 있는지 확인한다.
3. 다른 팀장이 없는 경우, 시스템은 탈퇴 액션을 차단하고 차단 메시지를 노출한다(예: 팀장 이양이 먼저 필요함을 알리는 수준의 안내 — 정확한 문구는 화면 설계 단계에서 결정).

**기대 결과**: 팀에 팀장이 0명이 되는 상태는 발생하지 않는다. 팀장 이양 UX 자체의 세부 절차는 범위 외(도메인 정의서 §10.2)이므로 이 문서에서 더 구체화하지 않는다.

---

### EX-02. 동일 일정에 대한 동시 수정 충돌 (BR-10 / §6.2)

**상황**: 팀장이 서로 다른 브라우저 탭(또는 세션) 두 곳에서 같은 일정의 수정 화면을 열어 각각 다른 내용으로 저장을 시도한다.

**흐름**
1. 팀장이 탭 1에서 일정 A의 시간을 14:00으로 변경하고 저장한다. → 시스템은 저장을 반영한다.
2. 같은 팀장이(먼저 열어두었던) 탭 2에서 일정 A의 제목을 수정하고 저장한다. → 시스템은 탭 2의 저장 요청을 나중에 도달한 수정으로 처리하여 반영한다.
3. 결과적으로 탭 2의 저장 내용이 최종 상태가 되며, 탭 1에서 저장했던 시간 변경 내용은 반영되지 않은 채 남는다(Last-Write-Wins).

**기대 결과**: 시스템은 별도의 충돌 경고 없이 나중 저장을 그대로 반영한다(BR-10 초안 가정). 충돌 경고 UI나 낙관적 잠금은 이번 범위에 포함하지 않는다(도메인 정의서 §10.2, PRD §3.2).

---

### EX-03. 다중 팀 소속 사용자가 통합 뷰를 기대하는 경우 (BR-11 / §6.3)

**상황**: 여러 팀에 소속된 사용자가 모든 소속 팀의 일정을 한 캘린더 화면에서 동시에 보고자 시도한다.

**흐름**
1. 사용자가 캘린더 화면에서 여러 팀의 일정을 한 번에 겹쳐 보려 하지만, 시스템은 팀 선택 UI를 통해 한 번에 하나의 팀만 선택할 수 있도록 제한한다(SC-09 참고).
2. 사용자가 다른 팀의 일정을 확인하려면 팀 전환 UI를 통해 화면 전체를 전환해야 한다.

**기대 결과**: 여러 팀의 일정을 하나의 캘린더에 병합해 보여주는 기능은 제공되지 않는다. 이는 미구현 버그가 아니라 BR-11에 따른 의도된 동작이다.

---

### EX-04. 마지막 소속 팀 탈퇴 (BR-12 / §6.4)

**상황**: 사용자가 자신이 유일하게 소속되어 있던 팀에서 탈퇴한다.

**흐름**
1. 사용자가 유일한 소속 팀에서 "팀 탈퇴"를 클릭하고 확정한다.
2. 시스템은 사용자를 해당 팀에서 제외 처리한다. 사용자 계정 자체는 삭제하지 않는다.
3. 이후 사용자가 다시 로그인하면, 시스템은 로그인은 정상적으로 허용하되 조회 가능한 팀 캘린더/채팅이 없는 상태(소속 팀 없음)로 화면을 표시한다.

**기대 결과**: 계정은 유지되고, 사용자는 신규 팀 생성 또는 다른 팀 재가입을 통해 다시 서비스를 이용할 수 있다.

---

## 4. 범위 외 시나리오 안내

다음 항목들은 도메인 정의서 §10.2 및 PRD §3.2에서 이미 범위 외로 규정되어 있으므로, 본 문서에서도 별도 시나리오로 다루지 않는다(중복 나열 대신 두 문서를 참조):

- 외부 캘린더/메신저 연동(Google Calendar, Slack, Outlook 등)
- ScheduleChangeRequest의 승인/반려/보류 등 정식 상태 전이 워크플로 및 처리 결과 알림
- 팀 초대·가입·팀장 이양의 상세 UX/절차
- 낙관적 잠금 등 정교한 동시성 제어
- 사용자 로컬 타임존 기준 채팅방 일자 구분
- 푸시/이메일 등 외부 알림 채널, 모바일 네이티브 앱, 오프라인 동기화
- 반복 일정, 접근성(A11y) 대응, 다국어/국제화
