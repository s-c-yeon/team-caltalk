## 1. 프로젝트 개요

**Team CalTalk**은 팀 단위로 일정을 관리하고, 캘린더와 채팅을 하나의 화면에서 함께 사용할 수 있는 팀 캘린더 협업 애플리케이션이다.

### 1.1 문제 정의

- 팀원 각자의 개인 캘린더, 메신저, 엑셀 등에 일정이 흩어져 있어 팀 전체 일정을 한눈에 파악하기 어렵고, 일정 충돌·누락이 자주 발생한다.
- 일정과 그 일정에 대한 논의(대화)가 서로 다른 도구에 분리되어 있어, 특정 일정이 왜 그렇게 정해졌는지 맥락을 되짚기 어렵다.
- 팀장이 팀원 개개인의 일정을 직접 통제하기 어려워, 팀 전체 일정 조율에 시간이 많이 소요된다.

### 1.2 목적

- 팀 단위 일정을 월/주/일 단위로 통합 관리하고 팀원 전체에 공유한다.
- 캘린더와 채팅을 같은 화면에서 제공하여, 일정 변경 논의와 그 맥락(이력)을 한곳에 남긴다.
- 일정에 대한 권한을 팀장에게 집중시켜 일관된 팀 일정 운영을 가능하게 하고, 팀원의 개별 요청은 채팅을 통한 "요청 → 팀장 수동 반영" 흐름으로 처리한다.

> **"요청-승인 흐름"의 의미 (범위 명확화)**: 본 문서에서 말하는 "요청-승인 흐름"은 대기/승인/반려 등의 상태를 갖는 정식 승인 워크플로(state machine)를 의미하지 않는다. 실제로는 (1) 팀원이 채팅 메시지 작성 시 "변경 요청" 액션을 명시적으로 선택해 요청을 남기고, (2) 팀장이 이를 검토한 뒤 필요하다고 판단하면 팀 일정을 직접 수정하는, **팀장의 수동 반영** 행위를 가리킨다. 요청 자체에는 별도의 승인/반려 상태값이 부여되지 않는다. 정식 상태 전이 워크플로가 필요한지 여부는 10.2절 범위 외 항목에 따라 상세 설계 단계에서 별도로 검토한다.

## 2. 핵심 용어 정의 (Ubiquitous Language)

| ID | 용어 | 정의 |
|---|---|---|
| TERM-01 | 사용자(User) | 인증(로그인)을 거쳐 시스템을 이용하는 주체. 하나 이상의 팀에 팀원 또는 팀장으로 소속될 수 있다. |
| TERM-02 | 팀(Team) | 일정과 채팅을 공유하는 사용자들의 집합. 정확히 한 명의 팀장을 가진다. |
| TERM-03 | 팀장(Team Leader) | 팀 일정에 대한 생성/수정/삭제 권한을 가진 팀 내 관리자 역할. |
| TERM-04 | 팀원(Team Member) | 팀 일정을 조회만 할 수 있는 역할. 자신과 관련된 일정 변경은 채팅의 "변경 요청" 액션을 통해 요청한다. |
| TERM-05 | 팀 일정(Team Schedule) | 팀 전체 또는 팀원 개인에게 배정되어 팀 캘린더에 표시되는 일정 항목. 월/주/일 단위 뷰로 조회된다. |
| TERM-06 | 채팅방(Chat Room) | 특정 날짜(일자)에 대응하여 대화가 이루어지는 공간. 날짜별로 이력이 분리되어 누적된다. |
| TERM-07 | 채팅 메시지(Chat Message) | 채팅방에 남기는 개별 대화 기록. 발신자, 시각, 내용을 가지며, 작성 시 일반 메시지 또는 "변경 요청" 메시지 중 하나로 명시적으로 구분되어 전송된다. |
| TERM-08 | 일정 변경 요청(ScheduleChangeRequest) | 팀원이 채팅 메시지를 작성할 때 **"변경 요청" 액션을 명시적으로 선택하여 전송한 경우에만** 생성되는 레코드. 대상 일정(신규/기존)에 대한 등록·수정·취소 요청 의도를 담으며, 원본이 된 채팅 메시지와 1:1로 연결된다. 자연어 의미 분석(NLP)을 통한 자동 판별은 하지 않는다. |
| TERM-09 | 채팅 이력(Chat History) | 날짜별로 누적되어 남는 채팅 메시지 기록 전체. |

> **용어 확정 사항**: 이전 초안에서 사용되던 "채팅 메시지 중 변경을 요청하는 의미를 갖는 항목"과 같은 모호한 표현은 전부 제거하고, TERM-08과 같이 **사용자의 명시적 액션(버튼/토글 등 UI 선택)을 트리거로 하는 결정론적 정의**로 확정한다. 즉, ScheduleChangeRequest는 오직 "변경 요청" 액션이 선택된 채팅 메시지에 대해서만, 그리고 그런 메시지에 대해서는 항상 생성된다. 자연어 내용을 분석해 요청 여부를 추론하는 방식은 채택하지 않는다.

## 3. 핵심 액터(역할) 정의

### 3.1 팀장 (Team Leader)
- 팀 일정의 생성(Create), 수정(Update), 삭제(Delete) 권한을 가진다.
- 팀원이 채팅으로 남긴 일정 변경 요청(TERM-08)을 검토하고, 필요 시 해당 일정을 직접 수정하여 반영한다(수동 반영이며, 별도의 승인/반려 상태 처리는 하지 않는다. 1.2절 참고).
- 팀 캘린더 및 채팅 이력을 조회할 수 있다.
- 한 팀에는 팀장이 1명 존재한다(초안 범위에서는 단일 팀장 가정).

### 3.2 팀원 (Team Member)
- 팀 일정을 조회(Read)만 할 수 있고, 직접 생성/수정/삭제할 수 없다.
- 자신과 관련된 일정 변경이 필요할 경우, 해당 일자 채팅방에서 메시지를 작성할 때 "변경 요청" 액션을 명시적으로 선택하여 팀장에게 요청한다.
- 팀 채팅에 참여하여 메시지를 주고받을 수 있다.

> 한 사용자가 팀 A에서는 팀장, 팀 B에서는 팀원일 수 있다 — 역할은 사용자 고정 속성이 아니라 팀 소속 관계에 종속된다.

### 3.3 팀 라이프사이클 (생성/가입/탈퇴) — 초안 수준 가정

이번 초안은 팀의 생성·가입·탈퇴에 대한 상세 워크플로(초대 승인, 이메일 인증 등)를 설계하지 않는다. 대신 이후 단계 설계의 출발점이 될 수 있도록 다음과 같은 합리적 가정만 명시한다.

- **팀 생성**: 사용자가 새 팀을 생성하면, 그 생성자가 해당 팀의 팀장이 된다(자동 임명). 한 팀은 생성 시점부터 항상 정확히 1명의 팀장을 가진다.
- **팀 가입**: 팀원은 팀장으로부터 공유받은 초대 코드/링크 등을 통해 팀에 합류한다고 가정한다. 초대 발급·승인의 구체적 절차(예: 팀장 승인 필요 여부)는 범위 외이며 상세 설계 단계에서 정한다.
- **팀장의 탈퇴**: 팀장도 원칙적으로 팀을 탈퇴할 수 있다고 가정하되, 탈퇴 시 팀에 팀장이 0명이 되는 상태를 허용하지 않는다. 구체적 처리 방식은 6.1절(예외 상황) BR-09를 참고한다.
- **팀원의 탈퇴**: 팀원은 언제든 특정 팀에서 탈퇴할 수 있으며, 탈퇴 시 해당 팀의 일정/채팅 접근 권한을 잃는다.
- 위 항목들의 상세 절차(초대 UX, 알림, 탈퇴 확인 단계 등)는 이번 초안의 범위 외이며, 10.2절에 따라 상세 설계 단계에서 다룬다.

## 4. 핵심 도메인 개념 및 관계

```
사용자(User) ──소속── 팀(Team)
   │(역할: 팀장/팀원)         │
   │                    1 ── N
   │                 팀 일정(Schedule)
   │                          │
   │                     N ── 1 (해당 일자)
   └───작성────► 채팅 메시지(ChatMessage) ── N:1 ── 채팅방(ChatRoom, 일자별)
                       │
              작성 시 "변경 요청" 액션을
                 명시적으로 선택한 경우에만
                       ▼
              일정 변경 요청(ScheduleChangeRequest)  ── 1:1 ── 원본 채팅 메시지
                       │
                (팀장이 검토 후 수동으로 일정에 반영, 상태값 없음)
```

- **사용자(User)**: 이메일 등으로 인증되는 계정. 하나 이상의 팀-역할 관계를 가진다.
- **팀(Team)**: 일정과 채팅의 공유 범위 단위. 팀장 1명과 다수의 팀원으로 구성된다.
- **팀 일정(Schedule)**: 팀에 종속되며 제목, 일시(시작/종료), 대상(팀 전체 또는 특정 팀원), 작성자(팀장)를 가진다. 월/주/일 캘린더 뷰에 표시되는 최소 단위 개념이다. 데이터 제약은 8절(데이터 제약 및 경계 조건)을 참고한다.
- **채팅방(ChatRoom)**: 팀과 특정 일자(날짜)의 조합으로 식별되는 대화 공간. 해당 일자의 일정에 관한 논의가 이루어진다.
- **채팅 메시지(ChatMessage)**: 채팅방에 속하며 작성자, 작성 시각, 본문을 가진다. 작성 시 "일반 메시지"와 "변경 요청 메시지" 중 하나로 명시적으로 구분되어 전송되며, 날짜별로 누적되어 채팅 이력을 구성한다.
- **일정 변경 요청(ScheduleChangeRequest)**: 팀원이 채팅 메시지 작성 시 "변경 요청" 액션을 **명시적으로 선택한 경우에 한해** 생성되는 레코드로, 원본 채팅 메시지와 1:1로 연결된다. 일정의 신규/수정/취소 중 어떤 유형의 요청인지는 팀원이 메시지 작성 UI에서 함께 선택한다고 가정한다(세부 입력 항목은 설계 단계에서 구체화). 팀장의 검토 후 실제 일정 반영 여부가 결정되며, 이 반영은 팀장이 팀 일정을 직접 수정하는 행위로 이루어진다 — 별도의 승인/반려 상태 모델은 두지 않는다(초안 범위, 1.2절·10.2절 참고).

## 5. 핵심 비즈니스 규칙

| ID | 규칙 |
|---|---|
| BR-01 | **인증 필수**: 모든 기능은 로그인한 사용자만 이용할 수 있다. |
| BR-02 | **팀 소속 필수**: 사용자는 특정 팀에 소속되어야 해당 팀의 일정/채팅에 접근할 수 있다. |
| BR-03 | **일정 CUD는 팀장 전용**: 팀 일정의 생성·수정·삭제는 해당 팀의 팀장만 수행할 수 있다. |
| BR-04 | **팀원은 조회만 가능**: 팀원은 팀 일정을 열람할 수 있으나 직접 변경할 수 없다. |
| BR-05 | **일정 변경 요청은 명시적 액션으로만 생성**: ScheduleChangeRequest는 팀원이 채팅 메시지 작성 시 "변경 요청" 액션을 명시적으로 선택하여 전송한 경우에만 생성된다. 메시지 본문 내용에 대한 자연어 의미 분석(NLP)으로 자동 판별하지 않는다. 요청의 최종 반영 여부는 팀장의 수동 판단에 따른다(상태 머신 없음). |
| BR-06 | **채팅은 일자 단위로 이력화**: 채팅 메시지는 특정 날짜의 채팅방에 귀속되며, 날짜별로 이력이 분리·보존된다. "일자" 구분 기준은 8절 데이터 제약을 따른다. |
| BR-07 | **캘린더-채팅 연동 조회**: 사용자는 캘린더에서 특정 날짜를 선택해 해당 일자의 채팅 이력을 함께 확인하며 일정 관련 논의 맥락을 파악할 수 있다. |
| BR-08 | **조회 범위**: 캘린더는 월/주/일 단위 뷰를 지원하며, 팀원은 자신이 소속된 팀의 일정만 조회할 수 있다. |
| BR-09 | **팀장 탈퇴/부재 처리**: 팀장은 팀에 다른 팀장이 지정되지 않은 상태에서는 탈퇴할 수 없다(팀장 0명 상태 금지). 팀장 이양(다른 팀원을 팀장으로 지정)이 선행되어야 탈퇴가 가능하다고 가정한다. 이양 절차의 UX 세부사항은 범위 외. |
| BR-10 | **동시 수정 충돌 처리**: 팀장이 동일 일정을 여러 세션에서 동시에 수정하는 경우, 나중에 저장 요청이 도달한 수정이 이전 수정을 덮어쓸 수 있다(단순 Last-Write-Wins을 초안 가정으로 명시). 낙관적 잠금(optimistic lock) 등 정교한 동시성 제어 방식의 도입 여부는 설계 단계에서 검토한다. |
| BR-11 | **다중 팀 소속 시 캘린더 조회 방식**: 사용자가 둘 이상의 팀에 소속된 경우, 캘린더는 팀 단위로 전환하여 조회하는 것을 기본으로 한다(한 화면에 여러 팀 일정을 동시에 병합 표시하지 않음). 팀 간 일정 병합 뷰 제공 여부는 범위 외. |
| BR-12 | **마지막 소속 팀 탈퇴 시 계정 상태**: 사용자가 마지막으로 남아있던 소속 팀에서 탈퇴하더라도 사용자 계정 자체는 삭제되지 않고 유지된다(로그인은 가능하나 접근 가능한 팀 일정/채팅이 없는 상태). 계정 비활성화 등 별도 처리는 범위 외. |

## 6. 예외 상황 (Exception Scenarios)

이전 초안에서는 예외 상황이 다뤄지지 않았다. 아래 항목은 최소한의 예외 케이스를 식별한 것이며, 각 항목의 상세 처리 절차(에러 메시지, 복구 UX 등)는 설계 단계에서 구체화한다.

### 6.1 팀장 탈퇴/부재 시 처리 (BR-09 관련)
- 팀장이 존재하지 않는 상태(0명)는 허용하지 않는다.
- 팀장이 탈퇴를 시도할 때 다른 팀원이 없다면(팀장 1인 팀), 팀 자체를 삭제하거나 탈퇴를 차단하는 두 가지 선택지가 있을 수 있다 — 이번 초안에서는 **탈퇴 차단(다른 팀원이 있어야 팀장 이양 후 탈퇴 가능)**을 기본 가정으로 둔다. 최종 정책은 설계 단계에서 확정한다.

### 6.2 동시 수정 충돌 (BR-10 관련)
- 동일 일정에 대해 팀장이 두 개 이상의 브라우저 탭/세션에서 동시에 수정을 시도하는 경우를 가정한다.
- 초안 가정: Last-Write-Wins(나중 저장이 이전 저장을 덮어씀). 데이터 정합성 보장을 위한 버전 관리/락 전략은 범위 외이며 상세 설계 단계에서 다룬다.

### 6.3 다중 팀 소속 시 캘린더 조회 (BR-11 관련)
- 사용자가 여러 팀에 소속된 경우, 한 번에 하나의 팀을 선택하여 그 팀의 캘린더만 조회하는 것을 기본 동작으로 가정한다.
- 여러 팀의 일정을 하나의 캘린더에 겹쳐서 보여주는 기능(멀티팀 통합 뷰)은 이번 초안에 포함하지 않는다.

### 6.4 마지막 소속 팀 탈퇴 시 계정 상태 (BR-12 관련)
- 사용자가 유일하게 속해 있던 팀에서 탈퇴하면, 계정은 삭제되지 않고 "소속 팀 없음" 상태로 유지된다고 가정한다.
- 이 상태의 사용자는 로그인은 가능하지만 캘린더/채팅에는 접근할 일정 데이터가 없다. 신규 팀 생성 또는 재가입을 통해 다시 이용할 수 있다.

## 7. 주요 유스케이스

| ID | 유스케이스 |
|---|---|
| UC-01 | 사용자가 로그인하여 자신이 속한 팀의 캘린더를 월/주/일 단위로 조회한다. |
| UC-02 | 팀장이 팀 일정을 새로 등록한다. |
| UC-03 | 팀장이 기존 팀 일정을 수정하거나 삭제한다. |
| UC-04 | 팀원이 캘린더에서 특정 날짜를 선택해 해당 일자의 채팅방을 열고 이력을 확인한다. |
| UC-05 | 팀원이 자신의 일정 변경이 필요하여, 채팅 메시지 작성 시 **"변경 요청" 액션을 명시적으로 선택**하여 메시지를 전송한다. 이 메시지는 일정 변경 요청(ScheduleChangeRequest)으로 생성되어 팀장에게 노출된다. |
| UC-06 | 팀장이 채팅방에서 팀원이 남긴 일정 변경 요청을 확인하고, 필요하다고 판단하면 해당 일정을 직접(수동으로) 수정한다. 요청 자체에는 별도의 승인/반려 상태가 부여되지 않는다. |
| UC-07 | 팀원이 캘린더와 채팅창을 동시에 보면서 특정 일자의 일정과 관련 대화를 함께 파악한다. |

## 10. 범위(Scope) 및 범위 외 (Out of Scope)

### 10.1 이번 초안의 범위
- 팀 단위 일정의 월/주/일 캘린더 조회 및 공유
- 팀장의 일정 생성/수정/삭제, 팀원의 일정 조회
- 일자별 채팅방 및 채팅 이력
- 채팅 메시지 작성 시 "변경 요청" 액션을 명시적으로 선택하는 방식의 일정 변경 요청(ScheduleChangeRequest) 개념 및 생성 규칙(BR-05)
- 사용자 인증(로그인) 기반 접근 제어
- 팀 생성 시 생성자의 팀장 자동 임명 등 최소한의 팀 라이프사이클 가정(3.3절)
- 최소 수준의 예외 상황 식별(6절) 및 데이터 경계 조건 정의(8절)

### 10.2 범위 외 (초안에서 다루지 않음)
- 외부 캘린더 연동(Google Calendar, Outlook 등)
- 결제/구독 등 과금 모델
- 다중 팀장 체계, 팀 내 세부 권한(부팀장 등) 커스터마이징
- **일정 변경 요청의 상태 전이(대기/승인/반려 등) 상세 워크플로 및 알림 정책** — 위 1.2절에서 정리한 대로, 이번 초안에서 "요청-승인 흐름"은 팀장의 수동 반영을 의미할 뿐, 아래와 같은 정식 상태 머신은 범위 외로 남긴다:
  - ScheduleChangeRequest에 대한 승인/반려/보류 등의 상태 필드
  - 요청자에게 처리 결과를 알려주는 알림
  - 요청 처리 이력(누가 언제 반영/반려했는지)의 별도 추적
- 팀 초대·가입·팀장 이양의 상세 UX 및 절차 (3.3절의 가정 수준을 넘어서는 부분)
- 동시 수정에 대한 낙관적 잠금 등 정교한 동시성 제어 메커니즘 (8절/6.2절의 Last-Write-Wins 가정을 넘어서는 부분)
- 사용자 로컬 타임존 기준 채팅방 일자 구분 (8절의 서버 UTC 기준 가정을 넘어서는 부분)
- 푸시 알림, 이메일 알림 등 외부 알림 채널
- 모바일 네이티브 앱, 오프라인 동기화
- 반복 일정(recurring schedule)의 상세 규칙

> 위 범위 외 항목은 이후 상세 요구사항 정의 및 설계 단계에서 필요 시 별도로 다룬다.
